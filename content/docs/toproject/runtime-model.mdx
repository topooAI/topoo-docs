## Purpose
This document defines the `what + why` of the project operating layer.

It covers runtime-facing concerns that are necessary for agentic execution but should not replace the project object model.

It exists because once a project becomes agentic, static object definitions are no longer enough. Something must explain how those objects are allowed to run.

## Core Position
Project is not runtime.

Runtime is a facet of project.

Project defines and organizes the execution contract.
Runtime expresses the current live operating state of that contract.

Both are required.

This distinction is easy to lose in agent-heavy systems, where runtime can become so visible that it starts to masquerade as the whole product.

## What The Operating Layer Covers
The operating layer turns project structure into executable behavior.

It must explicitly cover:
- execution mode
- commands and skills
- hooks and automation
- session continuity
- activity as operating evidence

This layer exists to answer a question the object model cannot answer by itself:

"Given this project structure, how is execution actually allowed to happen?"

## Execution Mode
Execution mode describes the operating policy under which the agent is allowed to act.

Examples:
- plan
- supervised execution
- autonomous execution
- paused / waiting

This is distinct from `Status`.

`Status` describes the work state.
`Execution Mode` describes permission and operating posture.

This separation matters because projects can be active while execution is paused, or planning can still be active while direct execution is intentionally constrained.

## Commands And Skills
Tasks alone are not enough to describe an agentic project.

The operating layer must also acknowledge reusable executable capabilities such as:
- commands
- skills
- workflows
- setup actions
- review actions

These act on project state but are not the same thing as project objects.

That is why they belong here rather than in the object model doc. They are capabilities, not root semantics.

## Hooks And Automation
An agentic project requires event-driven behavior.

This includes:
- pre-execution hooks
- post-execution hooks
- review hooks
- sync hooks
- scheduled jobs / automation

These behaviors shape how activity is produced and how execution is governed.

Without them, the system looks interactive but not operationally explainable.

## Session Continuity
Agentic execution is not a one-shot request/response system.

It requires continuity features such as:
- session resume
- context carry-forward
- compaction / summarization
- interruption recovery
- diagnostic / doctor flows

This is where agentic systems differ from simple request/response products. Continuity is not a nice-to-have; it is part of the runtime contract.

## Activity As Operating Evidence
Activity is not only a note stream.

It is operating evidence.

Activity may be generated by:
- user actions
- agent actions
- git events
- hooks
- review / approval events
- automation runs

This is why `Activity` remains a top-level object type in the project-centered model.

Activity sits at the seam between object model and operating layer. It is the record where runtime becomes inspectable.

## Execution Engine Position
A Claude Code-like system belongs at the execution engine layer, not at the project root.

It is best understood as:
- a code-domain execution engine
- a repo-aware agent runtime
- a command / tool / hook driven execution environment

Project owns the execution contract.
The operating layer constrains execution behavior.
The execution engine executes inside those constraints.

This separation is what lets us learn from Claude Code-like systems without letting them redefine the whole platform around one engine's affordances.
